# 09 — Adapter Design (Claude Code & Codex)

How un-punt plugs into two different agent platforms while keeping a single source of truth. Builds on `03-architecture.md`, which sketched adapters at one paragraph each; this is the concrete design.

---

## Bottom Line Up Front

1. **The two platforms have converged.** Both expose YAML-frontmatter skills, hooks with the same five event names (`SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `Stop`), regex tool matchers, MCP, and a project context file (`CLAUDE.md` / `AGENTS.md`). The lift to support both is small.
2. **~90% of un-punt is platform-agnostic.** The skill body, the `.un-punt/` markdown spec, the trust contract, refusal lists, ranking formula, and the golden-set scenarios all live in `core/` and are loaded verbatim by both adapters. Adapters wire and place files; they do not implement capture, ranking, or lifecycle logic.
3. **Two real divergences.** (a) Codex's `workspace-write` sandbox makes `.git/` read-only by default, so the disposition prompt has to warn the user about per-commit approval prompts. (b) Codex has no `${CLAUDE_PLUGIN_ROOT}` analog, so hook scripts can't reference paths outside the plugin — each adapter bundles its own copy.
4. **MVP ships zero hooks.** Skill loads via description match; `settings.json` `permissions.deny` handles refusal-path guards. Hook configs in §4.3 / §5.3 are documented as **Phase 2 optional**, to be added only if eval shows description-match alone misses too much.
5. **MVP is Claude Code only.** Codex ships in Phase 2. The Codex design exists here so Phase 1 doesn't paint us into a corner.

## 1. Repository layout

```text
un-punt/
├── core/                                # All the IP. Platform-agnostic.
│   ├── skill/
│   │   ├── SKILL.body.md                # Skill body, no frontmatter
│   │   ├── reference/                   # markdown-spec, contract template,
│   │   │                                # refusal lists, ranking, id-derivation,
│   │   │                                # disposition prompt, examples
│   │   └── snippets/                    # pre-flight, cold-start, lifecycle
│   ├── hooks/                           # Shell scripts; both adapters call these
│   │   ├── on-session-start.sh
│   │   ├── on-user-prompt.sh
│   │   ├── on-stop.sh
│   │   └── on-pre-tool-use.sh
│   └── golden-set/                      # 50 scenarios + scenario format spec
│
├── adapters/
│   ├── claude-code/
│   │   ├── .claude-plugin/plugin.json
│   │   ├── skills/un-punt/
│   │   │   ├── SKILL.md                 # Built artifact: frontmatter + core body
│   │   │   ├── reference/               # Copied from core/skill/reference/ at build
│   │   │   └── hooks/                   # Copied from core/hooks/ at build
│   │   ├── hooks/hooks.json
│   │   ├── settings.json                # Default permissions
│   │   ├── snippets/CLAUDE.md.fragment
│   │   └── README.md
│   │
│   └── codex/
│       ├── .codex-plugin/plugin.json
│       ├── skills/un-punt/
│       │   ├── SKILL.md                 # Built artifact (different frontmatter)
│       │   ├── agents/openai.yaml
│       │   ├── reference/               # Copied from core/skill/reference/
│       │   └── hooks/                   # Copied from core/hooks/
│       ├── hooks/hooks.json
│       ├── snippets/AGENTS.md.fragment
│       ├── snippets/config.toml.example
│       └── README.md
```

Build step: `SKILL.md` is generated by concatenating each adapter's frontmatter on top of `core/skill/SKILL.body.md`. `reference/` and `hooks/` are copied (not symlinked) into each adapter at build time so plugins remain self-contained on disk — neither platform documents an env var that lets a hook script reach outside the plugin root.

`core/build.sh` (canonical contract — ~10 lines):

```bash
#!/usr/bin/env bash
set -euo pipefail
ROOT="$(cd "$(dirname "$0")/.." && pwd)"
for adapter in claude-code codex; do
  out="$ROOT/adapters/$adapter/skills/un-punt"
  mkdir -p "$out"
  cat "$ROOT/adapters/$adapter/skills/un-punt/_frontmatter.yml" \
      "$ROOT/core/skill/SKILL.body.md" > "$out/SKILL.md"
  rsync -a --delete "$ROOT/core/skill/reference/" "$out/reference/"
  rsync -a --delete "$ROOT/core/hooks/"           "$out/hooks/"
done
```

Idempotent. Re-runnable. The CLI's `install` command invokes it before copying files into `~/.claude/skills/`.

## 2. What lives in `core/` vs adapters

| Concern | core/ | adapter/ |
|---|---|---|
| Skill body (capture rules, planning, execution, lifecycle) | ✅ | — |
| Skill frontmatter (name, description, allowed-tools, etc.) | — | ✅ Different fields per platform |
| `.un-punt/` data model | ✅ | — |
| Trust contract template, refusal lists, ranking, ID derivation | ✅ | — |
| Disposition prompt text, pre-flight check sequence | ✅ | — |
| Worked examples (capture, plan, execute) | ✅ | — |
| Hook shell scripts (the logic) | ✅ `core/hooks/` | — |
| Hook config (event names, matchers, paths) | — | ✅ |
| Plugin manifest, distribution, install/uninstall | — | ✅ |
| Default permissions / sandbox config | — | ✅ |
| `CLAUDE.md` / `AGENTS.md` user-paste fragment | — | ✅ |
| Golden-set scenarios + scenario format | ✅ | — |
| Golden-set runner | — | ✅ Per-platform CLI wrapper |

Test for "could this live in core?" — for the skill body, the answer is always yes. If you find yourself writing platform-aware logic in the skill body, split the rule into a platform-neutral statement and let the adapter's frontmatter or hook config encode the platform-specific bits.

## 3. The two platforms, side by side

| Concern | Claude Code | Codex |
|---|---|---|
| Project context file | `CLAUDE.md` (no formal cap) | `AGENTS.md` (32 KiB cap; `project_doc_max_bytes` configurable) |
| Skill location (project) | `.claude/skills/<name>/SKILL.md` | `.agents/skills/<name>/SKILL.md` |
| Skill location (user) | `~/.claude/skills/<name>/` | `~/.agents/skills/<name>/` |
| Frontmatter — required | `description` recommended (`name` defaults to dir) | `description` (`name` defaults to dir) |
| Frontmatter — extras | `allowed-tools`, `model`, `effort`, `disable-model-invocation`, `user-invocable`, `paths`, `argument-hint`, `arguments`, `context`, `agent`, `hooks`, `shell`, `when_to_use` | Sparse on `SKILL.md` (no `allowed-tools` / `when_to_use`); richer metadata in `agents/openai.yaml` |
| Implicit invocation | On by default; `disable-model-invocation: true` to disable | On by default; `allow_implicit_invocation: false` in `agents/openai.yaml` |
| Slash exposure | Skill auto-exposed as `/<skill-name>` | Same: `/<skill-name>` and `$<skill-name>` |
| Hook config location | Plugin `hooks/hooks.json`, or `settings.json` `hooks` key | Plugin `hooks/hooks.json`, `~/.codex/hooks.json`, or `[hooks]` in `config.toml` |
| Hook feature gate | None — always on | `[features] codex_hooks = true`; project hooks gated on project trust |
| Hook input JSON | `{hook_event_name, session_id, cwd, tool_name, tool_input, prompt, …}` | Same shape (turn-scoped events also include `turn_id`) |
| Hook block mechanism | Exit 2 + stderr; or JSON `permissionDecision: "deny"` | Identical |
| Hook context injection | JSON `additionalContext` in `hookSpecificOutput` | Identical |
| Plugin manifest | `.claude-plugin/plugin.json` | `.codex-plugin/plugin.json` |
| Plugin root env var | `${CLAUDE_PLUGIN_ROOT}` | None documented — scripts must self-locate |
| MCP config | Plugin `.mcp.json` or settings | `[mcp_servers.<id>]` in `config.toml` |
| File edit tool | `Edit` / `Write` | `apply_patch` |
| Sandbox model | Permission prompts (`default` / `acceptEdits` / `bypassPermissions`) | OS-level (seatbelt / bubblewrap / Windows Sandbox); `read-only` / `workspace-write` / `danger-full-access`, plus orthogonal `approval_policy` |
| `.git/` writable in default mode? | Yes (with first-use prompt) | **No** — `.git/` is read-only in `workspace-write` |
| Distribution | Mature: official Anthropic marketplace + 3rd-party `marketplace.json` | OpenAI plugin marketplace launched Mar 2026 (Slack/Figma/Notion partners); repo + user `marketplace.json` (`$REPO_ROOT/.agents/plugins/marketplace.json`, `~/.agents/plugins/marketplace.json`) |

Differences cluster in: distribution maturity, sandbox model, the edit tool name (`Edit/Write` → `apply_patch`), and Codex's feature-flag/trust gates. None of these change the skill body.

## 4. The Claude Code adapter

### 4.1 Plugin manifest

`adapters/claude-code/.claude-plugin/plugin.json` (MVP — no hooks reference):

```json
{
  "name": "un-punt",
  "version": "0.1.0",
  "description": "Captures deferrals during agent work, proposes verified cleanup passes at wrap-up.",
  "skills": "skills/"
}
```

Phase 2 (if hooks become necessary): add `"hooks": "hooks/hooks.json"` and ship the config from §4.3.

No MCP server. Built-in `Bash`, `Read`, `Edit`, `Write`, `Glob` cover every operation.

### 4.2 Skill

`adapters/claude-code/skills/un-punt/SKILL.md` (built artifact: this frontmatter prepended to `core/skill/SKILL.body.md`):

```markdown
---
name: un-punt
description: |
  Capture deferrals (TODOs, .skip tests, `as any`, missing edge cases) into
  `.un-punt/items/<id>.md` as you work. At wrap-up, propose a verified
  cleanup pass in the developer's working tree. Refuse > Flag > Fix.
when_to_use: |
  - User signals end-of-feature, end-of-day, or area switch
  - You defer something during normal work
  - User invokes /un-punt explicitly
  - First session after install (cold-start inventory)
allowed-tools:
  - Read
  - Write
  - Edit
  - Glob
  - Grep
  - Bash(git *)
  - Bash(rg *)
  - Bash(sha256sum *)
  - Bash(npm test *)
  - Bash(pnpm test *)
  - Bash(pytest *)
  - Bash(cargo test *)
  - Bash(go test *)
  - Bash(tsc *)
  - Bash(eslint *)
---

[core/skill/SKILL.body.md inlined here at build time]
```

Skill auto-exposes as `/un-punt`. We don't ship a separate `/cleanup` command file (commands have been merged into skills — `.claude/commands/*.md` still works but skills are recommended — and `cleanup` as a name is too generic to claim).

### 4.3 Hooks (Phase 2 — not shipped in MVP)

> **MVP ships zero hooks.** The skill loads via description match; refusal-path guards live in `settings.json` `permissions.deny` (§4.4). Add hooks only if Phase 0 / Phase 1 evidence shows description-match misses wrap-up signals or capture triggers. The config below is the shape we'd ship if we needed to.

`adapters/claude-code/hooks/hooks.json` (Phase 2):

```json
{
  "hooks": {
    "SessionStart": [
      { "matcher": "startup|resume",
        "hooks": [{ "type": "command",
                    "command": "${CLAUDE_PLUGIN_ROOT}/skills/un-punt/hooks/on-session-start.sh" }] }
    ],
    "UserPromptSubmit": [
      { "hooks": [{ "type": "command",
                    "command": "${CLAUDE_PLUGIN_ROOT}/skills/un-punt/hooks/on-user-prompt.sh" }] }
    ],
    "Stop": [
      { "hooks": [{ "type": "command",
                    "command": "${CLAUDE_PLUGIN_ROOT}/skills/un-punt/hooks/on-stop.sh" }] }
    ],
    "PreToolUse": [
      { "matcher": "Edit|Write",
        "hooks": [{ "type": "command",
                    "command": "${CLAUDE_PLUGIN_ROOT}/skills/un-punt/hooks/on-pre-tool-use.sh" }] }
    ]
  }
}
```

What each hook does:

- **`SessionStart`** — Counts `.un-punt/items/` via `rg`; emits `additionalContext` with open/planned/resolved totals. Read-only. (No INDEX.md — counts are computed on demand.)
- **`UserPromptSubmit`** — Pattern-matches the prompt for wrap-up signals ("done", "wrap up", "ship it", "good place to stop"). If matched and ≥N items exist, emits `additionalContext` reminding the agent of the skill. Does not block.
- **`Stop`** — Reinforces the wrap-up suggestion if `UserPromptSubmit` missed the signal.
- **`PreToolUse` (Edit|Write)** — Final guardrail; blocks writes to refusal-listed paths (`.env`, `*.pem`, `.git/objects/**`). Returns `permissionDecision: "deny"`.

Every hook behavior is also in the skill body. Hooks are deterministic backstops; if disabled, un-punt still works.

### 4.4 Default permissions

`adapters/claude-code/settings.json` (merged on install):

```json
{
  "permissions": {
    "allow": [
      "Bash(git status)", "Bash(git rev-parse *)", "Bash(git diff *)",
      "Bash(git log *)", "Bash(git branch *)",
      "Bash(rg *)", "Bash(sha256sum *)",
      "Bash(lstat *)", "Bash(realpath *)",
      "Read(.un-punt/**)", "Write(.un-punt/**)", "Edit(.un-punt/**)"
    ],
    "ask": [
      "Bash(git commit *)", "Bash(git reset *)", "Bash(git checkout *)"
    ],
    "deny": [
      "Bash(git push *)",
      "Read(.env*)", "Read(**/*.pem)", "Read(**/*.key)", "Read(**/*_secret*)",
      "Bash(curl *)", "Bash(wget *)", "Bash(nc *)"
    ]
  }
}
```

**Caveat (April 2026):** `permissions.deny` has documented bypasses in current Claude Code versions (see [`08-design-decisions.md`](08-design-decisions.md) decision 14): multi-line bash, > 50-subcommand pipelines (Adversa CVE class disclosed Mar 2026, patched in v2.1.90), project-vs-user precedence bugs. Treat `deny` rules as convenience, not a sandbox; the load-bearing refusals are skill-level. The `Bash(curl|wget|nc *)` denies are a defense-in-depth layer for the verifier-script tampering case ([`05-skill-brief.md`](05-skill-brief.md) §4) — they don't replace the skill-side denylist; they backstop it.

**Bypass-mode detection (skill-side):** the skill checks `CLAUDE_DANGEROUSLY_SKIP_PERMISSIONS` and the runtime permission-state probe at session start. If detected, the skill refuses to operate and emits a one-line message — un-punt requires the standard permission system because hooks behave unpredictably under bypass mode (Anthropic GH issues #39523 / #18846 / #41615). See [`07-risks-and-evals.md`](07-risks-and-evals.md) C8.

### 4.4.1 Install-time settings.json merge semantics

`un-punt install claude-code` must merge the adapter's `settings.json` `permissions` block into the user's existing `~/.claude/settings.json` rather than overwrite it. Algorithm:

1. Read existing `~/.claude/settings.json` (or `{}` if absent).
2. For each of `allow`, `ask`, `deny`: union the existing array with the adapter's array; deduplicate by exact string match (case-sensitive); preserve the user's relative order, append our additions at the end.
3. Other top-level keys (`hooks`, `model`, etc.) are left untouched.
4. Write the merged object back. Format: 2-space indent, trailing newline.
5. Track our additions in `~/.claude/un-punt-install.json` (a small manifest of which lines we added) so `uninstall` can reverse exactly what we added without touching the user's own entries.

Idempotent: re-running `install` is a no-op (every union is already a superset).

### 4.5 Install / uninstall

**Distribution decision: self-hosted marketplace in the same repo as the plugin source.** No third-party marketplace dependency; the repo doubles as both source and distribution.

The repo root contains `.claude-plugin/marketplace.json` declaring marketplace name `un-punt` with plugin `un-punt` (locations point at `adapters/claude-code/`).

```bash
# One-time: add the marketplace
/plugin marketplace add <org>/un-punt

# Install / uninstall
/plugin install un-punt@un-punt
/plugin uninstall un-punt@un-punt --scope user
```

The README's hero install command (`/plugin install un-punt@un-punt`) assumes the marketplace has already been added in the user's Claude Code session. The README's Install section spells out the marketplace-add step.

Manual fallback (no marketplace at all): `git clone <repo> && cp -r adapters/claude-code/skills/un-punt ~/.claude/skills/`.

### 4.6 CLAUDE.md fragment

`adapters/claude-code/snippets/CLAUDE.md.fragment` — optional paste into project's `CLAUDE.md`:

```markdown
A `un-punt` skill is installed. Invoke it when wrapping up work, when
deferring something, or when the user runs `/un-punt`. Items live in
`.un-punt/`. Refuse > Flag > Fix.
```

### 5.3 Hooks (Phase 2 — not shipped in MVP)

> Same MVP rule applies: Codex adapter ships zero hooks at first launch. The config below is documented for Phase 2.

`adapters/codex/hooks/hooks.json` (Phase 2) — same shape as Claude Code, with three diffs:

```json
{
  "hooks": {
    "SessionStart": [
      { "matcher": "startup|resume",
        "hooks": [{ "type": "command",
                    "command": "<resolved-at-install>/skills/un-punt/hooks/on-session-start.sh" }] }
    ],
    "UserPromptSubmit": [
      { "hooks": [{ "type": "command",
                    "command": "<resolved-at-install>/skills/un-punt/hooks/on-user-prompt.sh" }] }
    ],
    "Stop": [
      { "hooks": [{ "type": "command",
                    "command": "<resolved-at-install>/skills/un-punt/hooks/on-stop.sh" }] }
    ],
    "PreToolUse": [
      { "matcher": "^apply_patch$|^Bash$",
        "hooks": [{ "type": "command",
                    "command": "<resolved-at-install>/skills/un-punt/hooks/on-pre-tool-use.sh" }] }
    ]
  }
}
```

The diffs from the Claude Code hooks:

1. **`PreToolUse` matcher** is `^apply_patch$|^Bash$` (Codex's structured-edit tool is `apply_patch`, not `Edit`/`Write`).
2. **No `${CLAUDE_PLUGIN_ROOT}` analog.** Either the install step rewrites paths to absolute, or the adapter ships a small launcher script that uses `BASH_SOURCE` to self-locate.
3. **`SessionStart` matcher.** Codex docs don't publicly document the match-value for session source the way Claude Code does (`startup|resume|clear|compact`). Treat the matcher as best-effort — drop it if `codex resume` doesn't fire the hook (see open question 3 in §7). Codex hooks may also be configured inline as `[[hooks.PreToolUse]]` tables in `config.toml`; we ship `hooks/hooks.json` for parity with the Claude Code adapter.

### 5.4 Hook prerequisites

Codex hooks require `[features] codex_hooks = true` in user or project `config.toml`, plus project trust for `<repo>/.codex/` configs. The install README must:

1. Detect whether hooks are enabled; if not, prompt to add the feature flag.
2. Document the trust prompt the user will see on first project session.
3. State explicitly that the skill works without hooks (degraded but functional).

### 5.5 Approval / sandbox guidance

We don't override the user's `config.toml`. `adapters/codex/snippets/config.toml.example` documents the recommended preset:

```toml
# Recommended: codex --full-auto
#   approval_policy = on-request, sandbox_mode = workspace-write

[features]
codex_hooks = true

# In workspace-write, .git/ is read-only by default. The disposition prompt's
# commit options will trigger one approval prompt per batch — expected behavior.
# Pick option 4 ("leave uncommitted") for a no-prompt path.
```

## 6. The two real divergences

Everything above can be flat-mapped from one adapter to the other. These two cannot.

### 6.1 Codex `.git/` is read-only in `workspace-write`

Codex's default sandbox makes `.git/` read-only even though the rest of the workspace is writable. The disposition prompt's commit options will each trigger one approval prompt.

**Mitigation:** the shared skill body includes one paragraph explaining the prompt clearly and pointing users to option 4 (leave uncommitted) as the no-prompt path. Single source, both adapters benefit.

### 6.2 No `${CLAUDE_PLUGIN_ROOT}` analog in Codex

Codex doesn't document a runtime env var pointing at the installed plugin directory. Hook scripts that reference bundled assets must self-locate.

**Mitigation:** at build time we copy `core/hooks/` into each adapter's plugin tree. Hook commands in the manifest reference paths inside the plugin. For Codex, the install step either rewrites the manifest's command paths to absolute, or wraps each script in a 3-line launcher that uses `BASH_SOURCE` to find its own directory. Either way, scripts never escape the plugin root.

## 7. Open questions to verify before Phase 2

Each is a 30-minute spike; gating Codex adapter design.

1. Does Codex's `Stop` hook reliably fire after a multi-turn agent task in `--full-auto`?
2. Size limit on `additionalContext` in Codex hook output? (Claude Code documents a 25k-token re-attachment budget; Codex docs are silent.)
3. Does `codex resume` re-fire `SessionStart` hooks?
4. Does a user-scope skill in `~/.agents/skills/un-punt/` work in a project where the user hasn't granted project trust?
5. `apply_patch` vs `Bash(sed ...)` for editing `.un-punt/` files — which does Codex actually choose, and does our `PreToolUse` matcher cover both paths?
6. Does Codex honor `argument-hint` in skill frontmatter (cosmetic, affects polish)?

These are unknowns, not blockers. The Claude Code adapter ships first regardless; Phase 2 starts with these spikes.
