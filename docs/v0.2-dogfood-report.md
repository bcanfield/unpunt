> **Canonical source of truth** for the v0.1 → v0.2 dogfood arc (April–May 2026). Originally captured in the punt-board repo's `dogfood-log.md`; consolidated here so the un-punt repo stands alone. All probe outcomes, agent quotes, and metrics in this file are the authoritative record.

## TL;DR

We built un-punt v0.1 (markdown skill, no hooks). We dogfooded it on a real product build (punt-board, a public dashboard for `.un-punt/` directories). The skill was empirically dormant during normal coding work — never auto-loaded, never silently captured deferrals, never offered wrap-up sweeps. Cold-start (`/un-punt`) recovered everything once invoked, but the "always-on" promise was empirically false.

We re-architected v0.2 around **Decision #21**: SessionStart + PostToolUse + UserPromptSubmit hooks that route deterministic events to the agent without classifying content (preserving Decision 2). The skill body remained the IP; hooks became thin event-routing scripts. Cross-platform validation showed hooks are now standard across Cursor, Codex, Copilot, Gemini CLI — the v0.1 "hooks are Claude-Code-specific" assumption was empirically wrong.

We re-validated. The four failed v0.1 probes pass under v0.2. Real-time captures land at confidence 0.95 vs cold-start's 0.4 floor, with the agent catching long-tail signals (untyped parameters, mock implementations, hardcoded values) that the regex-based cold-start could not. Decision 2 ("agent is the engine") was strengthened by the dogfood, not weakened.

A `$HOME` path-expansion bug surfaced 2 days post-ship across a session boundary; diagnosed in 30 minutes, fixed in 5 LOC + a regression check. v0.2 is launch-ready for Claude Code; Cursor + Codex adapters defer to v0.2.x patches per explicit user choice.

## v0.1 dogfood — the foundation finding

### Setup

- **Subject repo**: `punt-board` (the dashboard product the un-punt brief specified — TS+Next.js frontend + Python+FastAPI backend in a pnpm workspace, deployed against the punt-board × un-punt dashboard spec).
- **Install state**: un-punt v0.1 installed via `packages/cli/run.sh install claude-code` — skill files at `~/.claude/skills/un-punt/`, permissions block merged into `~/.claude/settings.json`. No hooks (per Decision 13).
- **Methodology**: error-analysis-first per Hamel Husain — observe real failures during real work, don't generate synthetic test scenarios.

### Initial observation (the one that triggered the diagnostic probes)

After ~3 sessions of normal scaffolding + dashboard-zone work, user noticed `.un-punt/items/` was effectively empty despite the agent having written multiple deferrals during that work. Verbatim:

> *"un-punt is not capturing anything. there have been about a dozen todos / fixmes that havent been captured. The only initial thing it captured was the smoke where i explicitly mentioned writing a todo."*

That observation — the gap between agent behavior (writing TODOs as side effects of feature work) and un-punt output (empty `items/`) — was the load-bearing finding of the entire dogfood. Every probe below was run to diagnose root cause.

### v0.1 verdict

The cold-start path worked as designed. The "always-on capture" path did not. **Decision 13's bullet 2 ("auto-invocation works when the description is well-written") was empirically false** on a well-formed ~1,100-char description well within the 1,536-char budget. The Phase 2 hook contingency the build plan documented as a fallback (*"Add SessionStart / Stop hooks in Phase 2 only if eval shows description-match alone is unreliable"*) had triggered.

## v0.2 architecture — Decision #21

### Why hooks now make sense

Two empirical findings from the dogfood + Q1+Q2 research broke Decision 13's assumptions:

1. **Description-match auto-loading does not fire on coding-topic conversations.** Probes 1, 2, 7 confirmed across multiple session entries.
2. **Hooks are now the cross-platform standard primitive, not a Claude-Code lock-in.** Per [`research/Q2a-codex-analogues.md`](research/Q2a-codex-analogues.md), [`Q2b-cursor-analogues.md`](research/Q2b-cursor-analogues.md), [`Q2c-other-platforms.md`](research/Q2c-other-platforms.md): Cursor 1.7 (Sept 2025), Codex 0.124.0 (April 2026), Copilot CLI (Feb 2026), and Gemini CLI all shipped stable hook systems with Claude-Code-compatible JSON-stdin/JSON-stdout contracts.

### The architecture (Decision #21 in [`08-design-decisions.md`](08-design-decisions.md))

- **SessionStart hook** — fires on every session in a `.un-punt/`-bearing repo. Emits `additionalContext` containing skill activation reminder. Compensates for description-match auto-load failure.
- **PostToolUse hook** with matcher `Edit|Write|MultiEdit` — fires after every file write. Structural pre-filter on path + gitignore (no content classification — Decision 2 preserved). Emits `additionalContext` reminding the agent to inspect the diff for capture-eligible patterns and apply the skill body's rules.
- **UserPromptSubmit hook** — fires before the agent processes the user's prompt. Detects wrap-up trigger phrases (verbatim from skill body's *Suggestion rules* section). Emits `additionalContext` prompting the agent to offer a sweep at the right moment.

### Sketch (ii) — the right line

Per [`research/Q4a-classification-line.md`](research/Q4a-classification-line.md): the hook does **structural pre-filter** (which events fire — path checks, gitignore, etc.); the agent does **content classification** (what each event means — is this a real deferral? what type? what confidence?). The line: WHICH events fire = mechanical (hook owns it); WHAT the event means = interpretive (agent owns it).

The May 1 implementation drift (Sketch (iii) — regex pre-classification of file content) was the codified anti-pattern. The v0.2 hooks have **zero regex over file content**.

### What didn't change (Decision 1, 2, 4 preserved)

- **Markdown all the way down** — items, sweeps, contracts, lifecycle stay markdown. Hooks parse `.un-punt/items/` markdown for state queries, but Decision 1 prohibits SQLite as the *storage substrate*; hook-side parsing of the markdown substrate doesn't violate it.
- **Agent is the engine** — Sketch (ii) preserves all 6 of Decision 2's why-bullets. Hooks route events; the agent classifies content.
- **Skill is the IP** — the skill body at `core/skill/SKILL.body.md` remained the source of truth. Hooks are thin event-routing scripts (~280 LOC bash total).

### What was widened — long-tail trigger coverage (Q4b)

The skill body's trigger table grew from 9 to 14 rows + an explicit *"examples are not exhaustive"* framing line. New rows cover: trivial catches, ignored Promise rejections, Rust `todo!()`, Go `panic("not implemented")`, disabled lints, debug logs, hardcoded values, mock implementations, multi-line commented-out code blocks. The 6-type enum stays closed; only the trigger examples expanded.

A "Confidence promotion" subsection was added to the cold-start procedure documenting how to convert cold-start flags (at 0.4) into fix-eligible items (per-type thresholds). Q8a finding #3 dispositioned as fix-in-v0.2.

### What was deferred (per [`v0.2-followups.md`](v0.2-followups.md))

- Cursor adapter (~80 LOC, single afternoon)
- Codex adapter (~70 LOC, single afternoon)
- Frontmatter mitigations (`when_to_use`, `paths`)
- Sketch (iv) `type: "prompt"` hooks (escape hatch if Sketch (ii) compliance < 60%)
- Top-3-areas dedupe (cosmetic)
- Refused-section full 12-rule enumeration (cosmetic)
- Line-drift recheck mechanism (`last_verified_at:` field; needs PostToolUse hook to observe edits)

## v0.2.0-fix1 — the May 4 `$HOME` regression

### What happened

After v0.2 shipped on May 2 and Probes 9–12 passed, user resumed normal punt-board feature work on May 4: Zone 4 (3 components: refusal-density, captures-vs-closes, heat-calendar), Zone 6 (calibration history), supporting backend changes. ~3 commits across an active session. At end-of-feature said *"ship it."*

**Observed**: items count unchanged at 19 (zero new captures across many `Edit`/`Write`/`MultiEdit` calls). No wrap-up suggestion fired. `.un-punt/.hook-state/suggested-sessions` marker file did not exist. Same failure pattern as v0.1 Probes 4 + 6/8 — but with v0.2 architecture allegedly active.

### Diagnosis

Hook scripts on disk fine, settings.json had hook block, manifest v2, all install bookkeeping correct. The bug: the command strings in user `settings.json` contained literal `$HOME` (e.g., `bash $HOME/.claude/skills/un-punt/hooks/session-start.sh`). Claude Code's hook executor did not shell-expand `$HOME` at exec time — bash received the literal `$HOME/...` as the script path and silently failed to find it. Hooks fired but their scripts couldn't run.

### Why it worked May 2 but not May 4

Most likely environmental — the May 2 session had `$HOME` resolved at the right moment (possibly inherited from a parent shell context that expanded at session-start time); the May 4 session crossed a boundary that lost the expansion. Whether due to a Claude Code update or normal session lifecycle, the literal absolute path is the documented-safe pattern (per `plugin-dev:hook-development` skill examples).

### Fix

- **Live install patched in place**: `~/.claude/settings.json` and `~/.claude/un-punt-install.json` had `$HOME` → literal home dir substituted via Python one-shot. Backups left at `*.pre-home-fix.bak`.
- **`packages/cli/src/install.ts` updated** to substitute `$HOME` at install time for all future installs (commit `b3b1303`).
- **`scripts/validate-v0.2.sh` got a regression check** (commit `4096e6e`): fails fast if user settings.json has literal `$HOME` in hook commands.

### Re-validation

After Claude Code restart, agent's response to the diagnostic question:

> *"the un-punt skill is available (listed in the skills section and **the SessionStart hook reminded me of its rules**), but I haven't loaded its SKILL.md body into this conversation."*

— confirming the SessionStart hook fires and reaches the agent. Subsequent Edit produced a new captured item (items: 19 → 20). v0.2 fully working end-to-end.

### Lesson

Silent failure mode is the worst kind. The bug was invisible to the install-time smoke tests we did for Q6 because those tests didn't cross session boundaries with the actual Claude Code runtime. The `validate-v0.2.sh` regression check is the floor; future Q7-style validation needs to test the FULL session-boundary lifecycle, not just same-session continuity.

## Sketch (ii) compliance — load-bearing for v0.3 escape-hatch decision

Of 1 capture-eligible Edit during validation (Probe 10 + the post-fix Test B), 1 was captured. Compliance: **100% on observed events**. Above the 80% pass threshold; well above the 60% Sketch-(iv)-prompt-hook escape-hatch threshold. v0.3 prompt hooks not warranted by current evidence.

**Caveat**: small N. Real-world usage will produce a larger denominator over time; the metric should be revisited after natural multi-day usage to confirm the 100% holds. If post-launch real-world compliance drops below 80%, evaluate. If below 60%, escalate to Sketch (iv) per Q3b's deferred path (would require new decision-register entry superseding Decision 2 bullets 3 + 4).

## v0.2 known limitations carried into launch

- **Line-drift on cold-start items** — items reference `file:line` at capture time; codebase refactors invalidate. Q8a finding #5: deferred to v0.3 (additive `last_verified_at:` field + PostToolUse-hook-driven recheck).
- **Top-3-areas double-counts nested paths** — Q8a finding #2: deferred (cosmetic).
- **Refused-section walks 7 of 12 rules explicitly** — Q8a finding #4: deferred (cosmetic).
- **Cross-platform parity** — v0.2 is Claude Code only at launch; AGENTS.md primer ships as universal floor. Cursor + Codex full adapters in v0.2.x patches (~80 LOC each, single-afternoon work per Q5c future-lift estimates).
- **Bypass mode (`--dangerously-skip-permissions`)** silently disables hooks per Decision 14's GH-issues evidence. Bypass-mode users get the v0.1 experience; documented as known limitation. Not a bug — same architectural floor Claude Code applies to all hooks.
