# un-punt v0.3 roadmap

> **What v0.3 is**: the next major un-punt release after v0.2 ships and gathers real-world usage signal. v0.3 is **gated on v0.2 evidence** — most v0.3 work either responds to v0.2 issues that were known but deferred, or extends v0.2 capabilities into areas v0.2 explicitly did not address.
>
> **For v0.2 work** (pre-launch + small post-launch patches), see [`v0.2-followups.md`](v0.2-followups.md).
>
> **For dogfood evidence** that motivated each v0.3 item, see [`v0.2-dogfood-report.md`](v0.2-dogfood-report.md).

---

## When does v0.3 work start?

v0.3 is not on a fixed schedule. Three triggers can open it:

1. **Compliance trigger** — Sketch (ii) compliance (real-time capture rate during normal Edit/Write/MultiEdit work) drops below **60%** over multi-day real usage. Triggers V03-1 (Sketch (iv) escape hatch).
2. **Stability trigger** — v0.2 has been stable in real use for ~4–8 weeks with no critical bugs, AND v0.2.x patches have shipped. Triggers all four v0.3 items as the natural next major.
3. **External trigger** — a Claude Code update breaks v0.2's hook contract, OR a competitor ships a feature that requires v0.3-level capabilities to match. Triggers re-prioritization.

Until one of these fires, v0.2 stays in maintenance mode and v0.3 work is **deferred but documented** here so nothing's lost.

## v0.3 work items

### V03-1 — Sketch (iv) `type: "prompt"` hooks (the escape hatch)

- **Trigger**: real-world Sketch (ii) compliance < 60% over multi-day usage
- **Status**: gated; do NOT start until trigger fires
- **Effort**: ~2-day chunk
- **Source of truth**: [`research/Q3b-decision-2-reread.md`](research/Q3b-decision-2-reread.md), [`research/Q4a-classification-line.md`](research/Q4a-classification-line.md) (Sketch iv option)

**What it is**: replace v0.2's command-based `bash` hooks with `type: "prompt"` hooks where the model judges classification at each event using the agent's same Claude. Per Q1b research, supported on Stop, SubagentStop, UserPromptSubmit, PreToolUse.

**Why it's the escape hatch**: if the agent ignores `additionalContext` reminders (Sketch ii compliance gap), prompt hooks make the classification non-optional — the model invocation IS the classification. This is what the v0.1 dogfood probes proved was *needed* if v0.2's hooks weren't enough; v0.2 evidence so far (n=1 capture event, 100% compliance) suggests we don't need it yet.

**Why it's NOT the v0.2 default**: violates Decision 2 bullets 3 + 4 per Q3b — adds LLM cost (1 extra model call per Edit) and adds a prompt template to maintain. Decision-register entry must supersede those bullets with empirical compliance data as justification.

**Implementation sketch**:
1. Replace each `core/hooks/*.sh` with a JSON config registering `type: "prompt"` with the appropriate prompt
2. Each prompt embeds the relevant section of the skill body + hook event payload
3. Model returns structured JSON with capture decisions
4. Agent applies the JSON (writes items, fires suggestion, etc.)

**Implementation gates**:
- New decision-register entry (#22 or later) superseding Decision 2 bullets 3 + 4
- Cost analysis on observed compliance data showing prompt-hook overhead is acceptable
- Prompt-template maintenance plan (how the prompt stays in sync with the skill body)

### V03-2 — Line-drift recheck mechanism

- **Trigger**: stability trigger (V03-1 doesn't fire; v0.2 stable + lots of accumulated stale items)
- **Status**: open; ready to start when stability trigger fires
- **Effort**: ~80 LOC across SKILL body + hook + items markdown spec
- **Source of truth**: [`research/Q8a-minor-findings-disposition.md`](research/Q8a-minor-findings-disposition.md) (finding #5; deferred to v0.3 at user gate)

**Issue**: items reference `file:line` at capture time. Codebase refactors invalidate them. v0.1 dogfood Day 3 surfaced concrete examples — `frontend/src/lib/api.ts:49` no longer existed at line 49 after a Day 3 refactor. Items pointed at stale code.

**Why deferred to v0.3**: requires PostToolUse hook to recheck on file edits. v0.2 hook architecture had to ship first and prove stable; now it has.

**Implementation sketch**:
1. Add `last_verified_at:` field to item frontmatter (additive; backward compat via missing-field handling)
2. Extend PostToolUse hook to detect when a tracked file is modified
3. On detect: scan `.un-punt/items/` for items referencing this file; for each, append `verified` or `drifted` row to its lifecycle table; update `last_verified_at` if still valid
4. Sweep planning surfaces `drifted` items in their own section of `plan.md`
5. Cold-start updates set `last_verified_at` to current time on capture

**Edge cases to handle**:
- File renamed → all items reference stale path. Detect via git rename tracking? Or just mark drifted.
- File deleted → items reference nonexistent path. Mark drifted.
- Line number shifted (insertion above) → content at original line differs. Mark drifted.
- Refactor preserves content but moves location → grep for original content; if found, update `line:`. If not, mark drifted.

### V03-3 — Convert dogfood-log misses to formal golden-set scenarios

- **Trigger**: stability trigger
- **Status**: open
- **Effort**: ~3–4 hrs (one session)
- **Source of truth**: [`v0.2-dogfood-report.md`](v0.2-dogfood-report.md) "the 18 inert-window deferrals" + "captured-but-correctly-skipped quality wins" sections

**Issue**: `core/golden-set/` is currently mostly synthetic scenarios. The v0.1 + v0.2 dogfood produced real empirical patterns:
- 18 real-source-deferral patterns (the inert-window deferrals)
- 9 correctly-skipped non-deferrals (8 fixture-content + 1 meta-comment)
- The v0.2.0-fix1 silent-failure case (paths-not-resolving)
- Probe 10's high-confidence capture with bonus reasoning

**Why this matters**: future skill iterations need regression coverage against real-world inputs, not just imagined scenarios. A skill change that breaks one of the 18 known dogfood patterns should fail the eval before it ships.

**Implementation sketch**:
1. For each of the 18 real-source deferrals, create a YAML scenario at `core/golden-set/real-NNN.yaml` with the file/line/expected-type/expected-confidence
2. For each of the 9 correctly-skipped non-deferrals, create a YAML at `core/golden-set/nocap-real-NNN.yaml`
3. Add adversarial scenarios for the fix1 silent-failure case (hook with broken path; verify validate-v0.2.sh catches it)
4. Update eval harness (V03-4) to run against these

### V03-4 — Eval harness refresh for hook-driven captures

- **Trigger**: stability trigger; ideally lands alongside V03-3 since they share infrastructure
- **Status**: open
- **Effort**: ~1 day
- **Source of truth**: [`research/Q5c-architecture-decision.md`](research/Q5c-architecture-decision.md) implications

**Issue**: `packages/evals/` runs Claude Agent SDK scenarios that *seed* the skill body into the session at run time. They don't test the v0.2 hook layer (because Agent SDK doesn't go through Claude Code's hook executor). The eval harness validates the skill body's logic but not the activation/nudge layer that v0.2 added.

**Why this matters**: v0.2 changes can pass the eval and still fail in production (the May 4 `$HOME` bug is exactly this — eval would have passed, install was correct, but path resolution failed at hook exec time).

**Implementation sketch**:
1. Add a new test category: `hook-integration/` scenarios that exercise hook firing
2. Test each hook event with synthetic JSON payloads (smoke-test that hooks emit valid `additionalContext`)
3. Add a "full-stack" mode that runs Claude Code in test mode against a fixture repo and verifies items get captured
4. Add the `$HOME` regression check (already in `scripts/validate-v0.2.sh`) as an eval-level regression scenario

## What's NOT v0.3

For completeness, items explicitly NOT in v0.3 scope:

- **Multi-developer support / team aggregation** — Phase 3+ in original build plan; out of v0.3 scope
- **MCP server for team-aggregation surface** — Decision 15 punted unless team aggregation becomes primary use case
- **Eng-leader dashboard / debt-velocity over time** — Phase 4 in original build plan
- **Tier 2 / Tier 3 platform adapters (Copilot CLI, Aider)** — backlog (low ROI per platform)
- **Landing page / marketing collateral overhaul** — out of code-scope; if needed, separate work stream

These live in [`v0.2-followups.md`](v0.2-followups.md) Backlog section. Move here only if a trigger fires that promotes them.

## How to start v0.3 when a trigger fires

1. Update this doc's "When does v0.3 work start?" section with which trigger fired and when
2. Open a branch `v0.3-WIP`
3. Pick the first item to work on (V03-1 if compliance trigger; V03-2/3/4 if stability trigger)
4. Run a research session per [`v0.2-research-plan.md`](v0.2-research-plan.md) methodology to validate the implementation approach against v0.2 architecture (especially Decision 2 implications for V03-1)
5. After research session, follow Q5/Q6 pattern from v0.2 — enumerate options, pick, implement in chunks
6. Re-validate via punt-board re-dogfood OR an equivalent subject repo before tagging v0.3.0

The methodology codified during v0.2 research (in `v0.2-research-plan.md`) is the load-bearing process artifact — apply it verbatim for v0.3.
