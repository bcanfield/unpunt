# un-punt v0.3 roadmap

> **What v0.3 is**: the next major un-punt release after v0.2 ships and gathers real-world usage signal. v0.3 is **gated on v0.2 evidence** — most v0.3 work either responds to v0.2 issues that were known but deferred, or extends v0.2 capabilities into areas v0.2 explicitly did not address.
>
> **For v0.2 work** (pre-launch + small post-launch patches), see [`v0.2-followups.md`](v0.2-followups.md).
>
> **For dogfood evidence** that motivated each v0.3 item, see [`v0.2-dogfood-report.md`](v0.2-dogfood-report.md).

---

## v0.3 north star

The v0.2 dogfood + the user-feedback pass that immediately followed surfaced a **UX direction**: *minimal footprint per repo, agent-judgment-driven (not regex-driven), one global install that works across many repos with no per-repo overhead, skill body lean enough not to dilute the agent's signal.*

v0.2 was an **architectural** fix (hooks for activation). v0.3 is the **UX** fix: less rigidity, less footprint, more agent-trust.

Concretely, this north star reshapes how each v0.3 item is scoped:

| v0.3 dimension | What it pushes toward |
|---|---|
| **Vibes** | Trigger detection by model judgment, not regex (V03-7) |
| **Footprint** | AGENTS.md/CLAUDE.md opt-in detection rather than auto-write (V03-8); skill body ≤ ~400 lines (V03-6) |
| **Cross-repo** | One global install; per-repo `.un-punt/` auto-creates on first use (no manual setup) |
| **Trust** | Automated promotion via slash command + sweep-time integration (V03-5) — let agent do the agent work, not the user |

---

## When does v0.3 work start?

v0.3 is not on a fixed schedule. Four triggers can open it:

1. **Compliance trigger** — Sketch (ii) compliance (real-time capture rate during normal Edit/Write/MultiEdit work) drops below **60%** over multi-day real usage. Triggers V03-1 (Sketch (iv) escape hatch).
2. **Stability trigger** — v0.2 has been stable in real use for ~4–8 weeks with no critical bugs, AND v0.2.x patches have shipped. Triggers all v0.3 items as the natural next major.
3. **External trigger** — a Claude Code update breaks v0.2's hook contract, OR a competitor ships a feature that requires v0.3-level capabilities to match.
4. **UX-evidence trigger** — items that v0.2 dogfood + early adoption flagged as friction (the V03-5 / V03-6 / V03-7 / V03-8 cluster). Each can trigger independently of others.

Until one fires, v0.2 stays in maintenance mode and v0.3 work is **deferred but documented** here so nothing's lost.

## v0.3 work items

### V03-1 — Sketch (iv) `type: "prompt"` hooks (the full classification escape hatch)

- **Trigger**: real-world Sketch (ii) compliance < 60% over multi-day usage
- **Status**: gated; do NOT start until trigger fires
- **Effort**: ~2-day chunk
- **Source of truth**: [`research/Q3b-decision-2-reread.md`](research/Q3b-decision-2-reread.md), [`research/Q4a-classification-line.md`](research/Q4a-classification-line.md) (Sketch iv option)

**Scope**: replace v0.2's command-based PostToolUse hook with a `type: "prompt"` hook where the model judges classification at each Edit/Write/MultiEdit. Per Q1b research, supported on PreToolUse / PostToolUse / Stop / SubagentStop / UserPromptSubmit.

**Why it's the escape hatch**: if the agent ignores `additionalContext` reminders (Sketch ii compliance gap), prompt hooks make classification non-optional — the model invocation IS the classification. v0.2 evidence so far (n=1 capture event, 100% compliance) suggests we don't need it yet.

**Why it's NOT the v0.2 default**: violates Decision 2 bullets 3 + 4 per Q3b — adds LLM cost (1 extra model call per Edit) and adds a prompt template to maintain. Decision-register entry must supersede those bullets with empirical compliance data as justification.

**Note**: V03-7 is a smaller, more targeted cousin — applies the prompt-hook pattern only to UserPromptSubmit (wrap-up trigger detection). V03-7 is much lower stakes than V03-1 and could ship first independently.

### V03-2 — Line-drift recheck mechanism

- **Trigger**: stability trigger
- **Status**: open; ready to start when stability trigger fires
- **Effort**: ~80 LOC across SKILL body + hook + items markdown spec
- **Source of truth**: [`research/Q8a-minor-findings-disposition.md`](research/Q8a-minor-findings-disposition.md) (finding #5; deferred to v0.3 at user gate)

**Issue**: items reference `file:line` at capture time. Codebase refactors invalidate them. v0.1 dogfood Day 3 surfaced concrete examples — `frontend/src/lib/api.ts:49` no longer existed at line 49 after a Day 3 refactor.

**Implementation sketch**:
1. Add `last_verified_at:` field to item frontmatter (additive; backward compat via missing-field handling)
2. Extend PostToolUse hook to detect when a tracked file is modified
3. On detect: scan `.un-punt/items/` for items referencing this file; for each, append `verified` or `drifted` row to its lifecycle table; update `last_verified_at` if still valid
4. Sweep planning surfaces `drifted` items in their own section of `plan.md`
5. Cold-start updates set `last_verified_at` to current time on capture

**Edge cases**: file renamed (use git rename tracking?); file deleted; line number shifted; refactor preserves content but moves location.

### V03-3 — Convert dogfood-log misses to formal golden-set scenarios

- **Trigger**: stability trigger
- **Status**: open
- **Effort**: ~3–4 hrs (one session)
- **Source of truth**: [`v0.2-dogfood-report.md`](v0.2-dogfood-report.md) "the 18 inert-window deferrals" + "captured-but-correctly-skipped quality wins" sections

**Issue**: `core/golden-set/` is currently mostly synthetic scenarios. The v0.1 + v0.2 dogfood produced real empirical patterns:
- 18 real-source-deferral patterns (the inert-window deferrals)
- 9 correctly-skipped non-deferrals (8 fixture-content + 1 meta-comment)
- The v0.2.0-fix1 silent-failure case (paths-not-resolving)
- Probe 10's high-confidence capture with bonus reasoning

**Why this matters**: future skill iterations need regression coverage against real-world inputs, not just imagined scenarios. A skill change that breaks one of the 18 known dogfood patterns should fail the eval before it ships.

**Implementation sketch**: see V03-3 items in earlier draft; create `core/golden-set/real-NNN.yaml` per pattern.

### V03-4 — Eval harness refresh for hook-driven captures

- **Trigger**: stability trigger; ideally lands alongside V03-3 since they share infrastructure
- **Status**: open
- **Effort**: ~1 day
- **Source of truth**: [`research/Q5c-architecture-decision.md`](research/Q5c-architecture-decision.md) implications

**Issue**: `packages/evals/` runs Claude Agent SDK scenarios that *seed* the skill body into the session at run time. They don't test the v0.2 hook layer (because Agent SDK doesn't go through Claude Code's hook executor). The eval validates the skill body's logic but not the activation/nudge layer that v0.2 added.

**Why this matters**: v0.2 changes can pass the eval and still fail in production (the May 4 `$HOME` bug is exactly this — eval would have passed; install was correct; but path resolution failed at hook exec time).

**Implementation sketch**: synthetic-payload smoke tests for each hook + full-stack mode that runs Claude Code in test mode against a fixture repo + add the `$HOME` regression check from `validate-v0.2.sh` as an eval-level scenario.

### V03-5 — Automated promotion (cold-start items → fix-eligible)

- **Trigger**: UX-evidence trigger (already validated by v0.2 user feedback — manual procedure is too much work)
- **Status**: ready to start anytime; promote-to-immediate if user feedback after launch confirms
- **Effort**: ~80–100 LOC across SKILL body + slash-command + sweep-planning integration
- **Source of truth**: v0.2 user feedback (this conversation); [`v0.2-followups.md`](v0.2-followups.md) BL-6 (originally backlog, promoted to v0.3)

**Issue**: the v0.2 "Promoting cold-start items to fix-eligible" subsection in SKILL.body.md documents an 8-step manual procedure for converting a 0.4-confidence cold-start item into a fix-eligible item. That's too much manual work — the user has to do agent work (read item, evaluate intent + scope + test coverage, raise confidence, append lifecycle row, update timestamps).

**Two-part fix**:

1. **`/un-punt promote <id>` slash command** — agent reads the item, evaluates whether it's mechanical+single-file+well-scoped+has-test-coverage, and proposes promotion. User confirms with one keystroke. The 8-step procedure becomes the agent's job.

2. **Sweep-time integration** — at sweep planning time, the agent automatically considers cold-start items at 0.4 confidence and proposes promotion candidates *inline with the sweep plan*. *"Items 1–3 are mechanical and ready to promote. Promote and fix in this sweep?"* The documented manual procedure becomes a fallback, not the primary path.

**Why this is high-leverage**: real users encountered this gap immediately in the v0.2 dogfood. The current docs make the user do agent work; v0.3 makes the agent do agent work. Decision 2 ("agent is the engine") fully respected — this is the agent-driven version of the same procedure.

### V03-6 — Skill body conciseness pass

- **Trigger**: UX-evidence trigger (validated by v0.2 user feedback — wordiness costs tokens AND dilutes signal)
- **Status**: open; could ship as v0.2.1 since mechanical (no behavior change)
- **Effort**: ~2–3 hrs (one focused editing session)
- **Source of truth**: [`docs/05-skill-brief.md`](05-skill-brief.md) line 13 (*"Keep `SKILL.md` body under ~500 lines and split deeper reference material into sibling files one level deep"*)

**Issue**: current `core/skill/SKILL.body.md` is **541 lines** post v0.2 widening (with the new "Suggest when" framing line just added). Anthropic's skill authoring guidance recommends ≤500 lines. We're slightly over. Wordiness has two costs: more tokens loaded into the agent's context (cost), and more text the agent has to weight equally (dilutes signal on the load-bearing rules).

**Where the bloat is** (rough audit):
- 4 worked examples (lines ~63–178) — consider moving 2–3 to a `core/skill/reference/worked-examples.md` file. Worked examples are great pedagogy but progressive-disclosure puts them in the right place.
- 12-item anti-instructions section — some could merge or move to reference.
- Repetition between "Capture rules" + "Always-on disciplines" sections.
- Sweep planning section has duplicate detail with the worked planning example.

**Goal**: target 350–400 lines without losing rules. Move bigger pedagogical sections to `core/skill/reference/`. Tighten repetition. The agent loads body eagerly + reference on-demand per progressive disclosure; tightening the eager load is a free win on cost AND accuracy.

**Why this is more important than it looks**: v0.2 user feedback specifically called out wordiness as harmful: *"wordiness costs more but also makes things less accurate."* That's correct — LLM behavior degrades when load-bearing instructions dilute into a sea of qualifiers and examples.

### V03-7 — Prompt-based wrap-up trigger detection (UserPromptSubmit only)

- **Trigger**: UX-evidence trigger or stability trigger
- **Status**: open; targeted, smaller-scope cousin of V03-1
- **Effort**: ~50 LOC + a small decision-register entry
- **Source of truth**: v0.2 user feedback (this conversation); [`research/Q4a-classification-line.md`](research/Q4a-classification-line.md) Sketch (iv)

**Background**: the v0.2.0 immediate fix dropped the regex pre-filter from `core/hooks/user-prompt-submit.sh` so the hook fires on every prompt + agent judges via skill body. That's a direct, low-cost solution. But it has a noise cost: the agent processes the wrap-up reminder on every prompt, even when most prompts are clearly mid-task.

**v0.3 refinement**: replace the command-based UserPromptSubmit hook with a `type: "prompt"` hook that calls a small focused model with the user's prompt + a tight rubric and returns `{trigger: bool, reason: string}`. Hook conditionally injects the wrap-up reminder ONLY when `trigger: true`. The agent's main turn is cleaner (no "consider wrap-up?" reminder unless it actually is one).

**Why this is distinct from V03-1**:
- V03-1 is full classification at every PostToolUse — high cost, high blast radius
- V03-7 is targeted at trigger detection in UserPromptSubmit only — once per turn, small focused decision, much cheaper
- V03-7 doesn't violate Decision 2 the same way V03-1 does — the model isn't classifying CONTENT (deferral or not), it's evaluating user-intent meta-question (wrap-up moment or not). That's closer to event routing than to content classification.

**Constraints check (preliminary)**: Decision 2 bullets 3 ("no additional LLM cost") and 4 ("no prompt to maintain") both apply but to a much smaller surface. Worth a research session before shipping. Decision-register entry would supersede those bullets ONLY for UserPromptSubmit, not for the full classification path.

### V03-8 — AGENTS.md / CLAUDE.md opt-in detection (when cross-platform adapters land)

- **Trigger**: when Cursor or Codex adapter ships (gated on stability trigger + Cursor/Codex demand)
- **Status**: open; needed only when cross-platform adapters land
- **Effort**: ~30 LOC in CLI install + new `--with-primer` flag
- **Source of truth**: v0.2 user feedback (this conversation); ties to v0.3 north star (footprint)

**Background**: v0.2.0 immediate fix dropped the AGENTS.md auto-write entirely (per user feedback that AGENTS.md belongs to the user). The template still ships at `~/.claude/skills/un-punt/AGENTS.md.template` for users to copy manually.

**v0.3 refinement (when Cursor/Codex adapters land)**: those adapters need the AGENTS.md primer for graceful degradation when un-punt's hooks aren't available on a given install. So the primer becomes load-bearing for cross-platform users — but should still be opt-in:

- Add `--with-primer` flag to CLI install: `un-punt install claude-code --with-primer`
- Detect both `AGENTS.md` and `CLAUDE.md` (some projects use one, some the other, some both, some neither)
- If user passes `--with-primer`:
  - If neither file exists → write a 3-line pointer at `<cwd>/AGENTS.md`:
    ```
    <!-- un-punt -->
    This repo uses [un-punt](https://github.com/bcanfield/unpunt). Skill at
    ~/.claude/skills/un-punt/SKILL.md is the source of truth; un-punt's hooks
    fire it at session start. AGENTS.md template at
    ~/.claude/skills/un-punt/AGENTS.md.template if you want richer guidance.
    ```
  - If `CLAUDE.md` exists but `AGENTS.md` doesn't → recommend the user add the 3-line pointer to `CLAUDE.md` (don't write `AGENTS.md`)
  - If both exist → recommend; don't write
  - If `AGENTS.md` exists → recommend; don't write
- **Default behavior (no flag)**: don't write anything. Match v0.2.0 stance.

**Why opt-in not opt-out**: AGENTS.md belongs to the user. Users who want un-punt to participate in their AGENTS.md are explicit about it; users who don't are protected by default. The v0.2 user feedback was clear on this.

### V03-9 — Windows / cross-platform shell support

- **Trigger**: real Windows-user demand signal (post-launch)
- **Status**: open; gated on demand
- **Effort**: ~1–2 days
- **Source of truth**: v0.2 user feedback (this conversation — bash availability question)

**Issue**: v0.2 hooks are bash-based. Mac/Linux: bash universal. Windows: bash is available via Git Bash or WSL but not guaranteed in PowerShell context. v0.2 README documents Mac/Linux as the supported surface; Windows users currently can't use v0.2 cleanly.

**Two options**:
1. **Convert hook scripts to Node.js** — Node is required by Claude Code itself, so it's universal. ~150 LOC across the 3 scripts. More verbose than bash but cross-platform.
2. **PowerShell + bash dispatch wrapper** — install detects platform; on Windows, ships PowerShell scripts that fall through to Git Bash where available, or pure-PowerShell otherwise.

**Trade-off**: Option 1 is cleaner long-term but more rewrite work. Option 2 keeps the bash scripts but adds a thin Windows wrapper. Decision: gate on actual demand. Most Claude Code adoption is Mac/Linux dev environments; Windows can wait until we have user demand.

### V03-10 — `PreToolUse` refusal-floor hook for bypass-mode safety

- **Trigger**: stability trigger (post-v0.2.x adapters); informed by the v0.2.0 bypass-mode correction
- **Status**: open
- **Effort**: ~50 LOC + decision-register entry
- **Source of truth**: v0.2.0 immediate corrections fact-check (current Claude Code docs at https://code.claude.com/docs/en/hooks-guide.md)

**Background**: per the v0.2.0 immediate-corrections fact-check, `PreToolUse` hooks specifically fire **before** any permission-mode check. A hook returning `permissionDecision: "deny"` blocks the tool **even in `bypassPermissions` mode**. This is a stronger guarantee than `permissions.deny` provides, because the latter becomes unreliable in bypass mode.

**Issue**: v0.2 ships SessionStart + PostToolUse + UserPromptSubmit but NO PreToolUse hook. The categorical-refusal floor (auth/payments/migrations/lockfiles/etc.) currently relies on `permissions.deny` + skill-body interpretive refusals. In bypass mode, only the skill-body interpretive layer remains — empirically ~5% miss rate per ADR #20.

**v0.3 addition**: ship a `PreToolUse` hook that enforces categorical refusals for auth/payments/migrations/lockfiles/etc. paths. The hook returns `permissionDecision: "deny"` when the agent attempts to edit/write these paths. **This is a stronger floor in bypass mode than v0.2 provides**, because PreToolUse fires before permission-mode-check.

**Why deferred to v0.3 (not v0.2.0)**:
- Adds another hook to the install pattern (more surface area to test)
- Path-pattern logic for auth/payments/etc. needs careful design — can't false-positive on legitimate auth library refactors
- v0.2 disposition prompt + skill body are sufficient for v0.2.0 launch (user said yes to "skill operates normally in bypass" stance per Decision 14)
- v0.3 stability trigger is the natural moment to add this safety lift

**Implementation sketch**: per-path-pattern PreToolUse hook with matchers for `auth/`, `oauth/`, `permission/`, `acl/`, `rbac/`, `crypto`, `payment/`, `billing/`, migrations, lockfiles, `__generated__/`, `.github/workflows/`. Each path-match returns `{permissionDecision: "deny", message: "un-punt categorical refusal: <rule>"}`.

## How to start v0.3 when a trigger fires

1. Update this doc's "When does v0.3 work start?" section with which trigger fired and when
2. Open a branch `v0.3-WIP`
3. Pick the first item to work on (compliance trigger → V03-1; UX trigger → V03-5/6/7/8; stability trigger → V03-2/3/4/10)
4. Run a research session per [`v0.2-research-plan.md`](v0.2-research-plan.md) methodology to validate the implementation approach against v0.2 architecture (especially Decision 2 implications for V03-1 and V03-7)
5. After research session, follow Q5/Q6 pattern from v0.2 — enumerate options, pick, implement in chunks
6. Re-validate via punt-board re-dogfood OR an equivalent subject repo before tagging v0.3.0

The methodology codified during v0.2 research (in `v0.2-research-plan.md`) is the load-bearing process artifact — apply it verbatim for v0.3.
